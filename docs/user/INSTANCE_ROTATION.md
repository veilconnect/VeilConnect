# 实例轮换手册（抗封锁 · 运维方）

> 面向**部署/运维者**：当某个域名或 IP 被封锁时，如何快速换一个继续提供服务。
> 终端用户在受限网络里怎么用，见 [`CENSORED_REGION_GUIDE.md`](CENSORED_REGION_GUIDE.md)。

## 核心思路：别守一个大目标，做「许多小而短命的实例」

公开的 `veilconnect.org` 是固定、显眼、最易被封的单点。VeilConnect 的真正抗封锁优势是**可自部署**：
让信任网络里的人各自在**未被标记的域名 + 干净 IP** 上跑一个小实例，封了就换。封锁方要追着打许多
分散、随时更换的实例，成本远高于封一个固定站点。

把 `veilconnect.org` 当作**公开演示/漏斗**（它会被封，无所谓），真正的受限区用户引导到私有自部署实例。

## 一条命令拉起一个一次性实例

```bash
# 有域名（自动 Let's Encrypt HTTPS）——先把 A 记录指向 VPS 公网 IP
scripts/spin-instance.sh up root@<VPS_IP> chat.example.com

# 没有域名（自动用 <公网IP>.sslip.io，仍是真证书）
scripts/spin-instance.sh up root@<VPS_IP>
```

脚本做了什么：检测/安装 Docker → rsync 构建上下文到远端（**不**上传 `.env`/密钥）→ 在远端跑
`scripts/install.sh`（在**远端**生成 TURN/IP 指纹等密钥，权限 600，本机不接触）→ 自动签证书、起栈、健康检查。

拆除（封了或不用了，彻底清数据卷）：

```bash
scripts/spin-instance.sh down root@<VPS_IP>
scripts/spin-instance.sh status root@<VPS_IP>   # 查看状态
```

## 轮换节奏（被封时）

1. **备好一个域名池**：提前注册若干互不关联的域名（不同注册商/不同命名风格，别用 veil/chat/vpn 等显眼词）。
   也备 2–3 台不同机房/不同 ASN 的干净 VPS。
2. **封了就换**：`down` 旧实例 → 选下一个域名 + 下一台 VPS → `up`。几分钟内换好。
3. **带外分发新链接**：把新地址通过**已建立的私密信道**（当面、已有的加密群、上一条仍可用的 VeilConnect 房间）
   发给用户。**绝不公开张贴**新域名——公开即等于把它送给封锁方的关键词爬虫。
4. **拆掉旧的**：避免留下可被探测/取证的残留。

## 让实例更难被「指纹识别」

- **TURN over 443/TLS**：在 `.env` 设 `TURN_TRANSPORT=tcp` 并尽量让中继走 443，使流量更像普通 HTTPS，
  少用奇怪端口暴露 STUN/TURN 特征。
- **中性域名**：用看起来普通的域名，别让域名本身成为封锁关键词。
- **别共用基础设施**：不同实例用不同 IP/证书/域名，互不关联——一个被关联识别不致连坐其他。
- **关掉详细日志**（默认即关）：`SIGNAL_VERBOSE` 保持 0，实例本身不沉淀「谁在何时连谁」的元数据。

## 监测封锁状态

- **OONI**（ooni.org）：检测并长期追踪你的域名在各地是否被封、被封方式（DNS 污染 / SNI 阻断 / TCP 重置）。
  可提交你的域名进入测试列表。
- 配合少量境内探针/志愿者，确认实例从受限网络的真实可达性，而不是只看自己这边能开。

## 边界与风险（务必如实告知用户）

- **别宣称「不可封锁」**。这是降低「打不开网页」这一最脆弱环节的概率，不是绝对绕过。过度承诺会害到真实用户。
- 运营面向受限地区的绕过基础设施，对**运维者**有现实的法律/合规风险；用户在境内使用绕过工具本身也可能被追责。
- 配合 PWA 离线缓存（装一次后界面可离线启动）与「用户把 VeilConnect 叠在 Tor/VPN/代理之上跑」一起用，
  比单独换域名更稳。详见 [`CENSORED_REGION_GUIDE.md`](CENSORED_REGION_GUIDE.md)。
