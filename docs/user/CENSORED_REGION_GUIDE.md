# 在受限网络中使用 VeilConnect（终端用户）

> 面向**在审查/封锁网络里的用户**。运维方如何轮换实例见 [`INSTANCE_ROTATION.md`](INSTANCE_ROTATION.md)。
>
> **先说实话**：VeilConnect 不是「翻墙工具」，也**不保证不可封锁**。它能做的是把通信端到端加密、隐藏双方
> 真实 IP，并在域名被封后仍能离线启动界面。能否连通，仍取决于信令/中继是否穿得过封锁。请据此评估风险。

## 一、趁能打开时，先把它「装下来」（PWA 离线缓存）

VeilConnect 是网页应用，支持离线优先缓存：**只要在域名还能访问时打开过一次**，应用界面（HTML/脚本/加密 Worker）
就被缓存到本地；之后即使该域名被 DNS 污染或 SNI 阻断，**界面仍能离线启动**——只剩「连接对方」这一步需要网络。

怎么装：

- **手机/桌面 Chrome、Edge**：打开站点 → 菜单 →「安装应用 / 添加到主屏幕」。之后从图标启动，无需再访问网址。
- **iOS Safari**：分享 →「添加到主屏幕」。
- 装好后断网打开，应能看到解锁/口令界面（能离线启动即说明缓存成功）。

> 注意：离线只保证**界面**能起来；真正发消息仍需信令与中继可达（见下一节）。

## 二、连不上时：把 VeilConnect 叠在已有的绕过工具之上

如果站点打不开或建连失败，最稳的办法不是等，而是**先有一条通往外网的隧道，再在其上跑 VeilConnect**：

1. 先连上你已有的 **VPN / Tor 浏览器 / 代理（如 Xray/Shadowsocks 客户端）**。
2. 在该隧道下用浏览器打开 VeilConnect（或它的当前镜像）。
3. 正常建房/加入。此时网页加载、信令、中继都走隧道，绕开本地封锁。

VeilConnect 负责**端到端加密 + 隐藏 IP + 不留服务器端聊天内容**；隧道负责**穿透封锁**。两者叠加，各司其职。

## 三、找到「当前可用」的地址

公开的 `veilconnect.org` 可能已被封。受限区用户应使用**私有自部署实例**或**镜像**：

- 通过**带外渠道**获取当前地址：可信的人当面给、已有的加密群、上一条仍可用的 VeilConnect 房间链接。
- 一个地址失效了，向你的联系人要新的——运维方会轮换域名（见运维手册）。
- **不要**在公开论坛/社交媒体搜或转发这些地址：公开传播会加速它被封。

## 四、每次连接都要核对 SAS（防中间人）——别跳过

信令服务器被视为**不可信中继**。建连后双方会看到一段**短验证码（SAS / 安全短语）**：

- 通过**另一条你信任的渠道**（电话、当面、已验证的其他加密通道）口头核对这段码两端是否一致。
- **一致才继续聊**。不一致 = 可能有人在中间替换密钥，立即停止。
- 这一步是你对抗 MITM 的最后防线，比「能连上」更重要。

## 五、保持这些隐私默认（别为了能连就关掉）

- **强制中继（relayOnly）默认开启**：只走 TURN 中继，隐藏你和对方的真实 IP。除非本地联调，**别**设
  `localStorage.vc.relayOnly='0'`——关掉会暴露你的 IP 给对端。
- **身份口令**：身份私钥用你的口令加密存在本地（PBKDF2 60 万轮 + AES-256-GCM）。用强口令；共用设备用完登出。
- **一次性用法**：敏感场景可用一次性身份、聊完清除；持久化房间只保留入口、**不**保存聊天内容，但谁拿到
  链接+口令以后仍能进，按需谨慎分享。

## 六、风险与边界（请认真读）

- **不是不可封锁**：这是降低被封后「连不上」概率的一套办法，不是绝对绕过。
- **信令元数据**：服务器看不到聊天**内容**，但 WebRTC 建连过程不可避免要经过信令/中继。默认配置已尽量少留
  元数据（不记 clientId/房间/明文 IP），但「你在用它」这件事在网络层并非完全不可观察。叠加 Tor/VPN 可进一步降低暴露。
- **法律风险**：在某些地区，使用绕过/加密通信工具本身可能带来风险。请结合自身处境评估，必要时优先用你已经
  信任并验证过的工具。VeilConnect 不会、也无法替你承担这种判断。
